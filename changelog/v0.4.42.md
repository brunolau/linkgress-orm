# v0.4.42 (unreleased)

## Fixes — driver parity (postgres.js / Bun clients)

The jest suite was only ever green on `PgClient`. Multi-statement-capable
clients (PostgresClient, BunClient) took two additional execution paths with
long-standing correctness bugs, now fixed — the full suite passes on all
three drivers, under Node and Bun:

- **Fully-optimized single-round-trip path dropped collection features**: the
  naive per-collection SQL in `executeFullyOptimized` ignored collection
  `.where()` filters, limit/offset, scalar aggregations, `firstOrDefault`,
  DISTINCT, constant-FK (`__LIT:`/SCD2) predicates, navigation paths, nested
  selectors, and custom mappers — silently returning wrong data (e.g. a
  filtered collection came back unfiltered). The fast path is now guarded by
  `isNaiveCollectionFastPathSafe`: only plain flat lists over a single-column
  `id` correlation whose column types are jsonb-stable (int/bool/text/uuid/
  float/json) qualify; everything else falls back to the two-phase path.
- **Temptable multi-statement mode diverged from every other strategy**: it
  fetched raw rows and grouped client-side, leaking driver-native values
  (numeric as string, timestamps as Date, `a__b` flattened nested columns) and
  breaking cross-strategy equality. It now emits the same server-side
  aggregation SQL as the legacy path (json_agg / scalar aggregates) — one
  round trip, identical values. Also fixed the ROW_NUMBER pagination wrapper
  referencing ORDER BY columns not projected by its subquery (42703 on
  `firstOrDefault` with ordering).
- **`querySimple`/`querySimpleMulti` mis-shaped results**: for a single
  statement, porsager/Bun `.simple()` returns the result set itself, which was
  treated as an array OF result sets (broke single-`FutureQuery` batches —
  `futureCount()` returned 0). Both clients now normalize via element-shape
  detection, and `querySimple` returns the last ROW-BEARING result set instead
  of a trailing DROP's empty set (which silently emptied temptable collections
  on Bun).
- **BunClient could never construct from config**: it required the
  non-existent `bun:sql` module; now resolves the `Bun.SQL` global (fallback
  `require('bun')`).
- **PostgresClient under Bun misreported "postgres package not installed"**:
  ESM-namespace `require()` results are unwrapped, and pool-construction
  errors propagate honestly.
- **ESM consumers could not load the package**: 67 type-only names were
  re-exported as values from the barrels (fatal for Bun/ESM linkers); now
  `export type`.

## Bun support

- New `DatabaseClient.supportsBinaryArrayResults()` capability: Bun.SQL
  ≤ 1.3.14 panics ("incorrect alignment") or produces object-shaped arrays
  when decoding native ARRAY result columns in binary (prepared) mode. When
  false, all collection strategies emit `json_agg` instead of `array_agg` so
  no native array reaches the wire. Standalone upstream repro:
  `debug/bun-sql-binary-array-repro.ts`.
- **`BunClient({..., prepare: false})` text-results mode**: unnamed statements
  make Bun use TEXT-format results, whose array decoding is correct — the
  panic surface disappears entirely (raw SQL included) and
  `supportsBinaryArrayResults()` auto-reports true, keeping `array_agg`.
  Object params are pre-stringified by the client (Bun's text mode would send
  them as "[object Object]"). Costs ~0.05–0.08 ms/query re-parse vs prepared
  mode; still faster than postgres.js on reads. The full jest suite passes in
  BOTH modes (`LINKGRESS_TEST_BUN_PREPARE=false` runs the suite in text mode).
- BunClient no longer copies result sets (`Array.from` dropped — Bun result
  sets are real arrays): −17% on 1000-row reads, −20% on jsonb-heavy reads,
  −30% on 2-statement transactions in the driver micro-bench.
- New BunClient option **`datesAsStrings`**: converts Date result values to
  PostgreSQL-text strings ('YYYY-MM-DD HH:MM:SS.mmm' UTC; exactly-midnight
  values become 'YYYY-MM-DD'). Bun's SQL client has no custom type-parser
  hook — this restores the timestamps-as-strings contract that apps configure
  via postgres.js `types` parsers.
- **`array()` custom type now serializes to PG array literals** ('{1,2,3}',
  quoted/escaped strings, nested arrays, NULLs) instead of passing raw JS
  arrays to the driver. Literal strings bind on every driver — raw JS arrays
  break BunClient (JSON-serialized: 08P01 in binary mode, brace-less text in
  text mode).
- **`DbCteBuilder` accepts an optional `client`**: CTE bodies are built
  eagerly at definition time, so array-producing aggregations inside them
  (toNumberList/toStringList) could not see the driver capability and always
  emitted array_agg. With a client passed, they emit json_agg for drivers
  without native-array support. Omitting the client keeps previous behavior.

## Performance — entity result mapping

- **`mapResultToEntity` no longer rebuilds column configs per row**: the
  no-selector read path (`toList()` without `.select()`) ran
  `Object.entries(schema.columns)` + `ColumnBuilder.build()` for every column
  of every row (~30k config constructions on a 3000-row × 10-column read — the
  largest ORM-side entry in the CPU profile). The property/column/mapper plan
  is now computed once per schema (WeakMap-cached) and the per-result-set
  present-column subset is derived from the first row, removing per-cell `in`
  checks. ORM mapping overhead on that read shape dropped from +2.2 ms to
  +0.5 ms over raw SQL (−76%); selector-based reads were already on the
  pre-analyzed path and are unaffected.

## Fixes — pg / postgres clients

- **PgClient's `useBinaryProtocol` mapping removed**: it mapped to pg's
  `rowMode: 'array'`, which is NOT the binary protocol — it switches rows to
  positional arrays and would corrupt every column-name-based mapping if
  enabled. `supportsBinaryProtocol()` now honestly returns false.
- **PgClient ESM-interop + honest errors**: `require('pg')` namespace shapes
  are unwrapped, and pool-construction errors propagate instead of
  masquerading as "install pg" (same fix PostgresClient received).
- **`PostgresClient.connect()` now reserves a real connection**
  (`sql.reserve()`, postgres >= 3.4) so the `PooledConnection` contract — one
  pinned session for temp tables / SET / advisory locks — actually holds;
  `release()` returns it to the pool. Falls back to the shared pool on older
  driver versions.
- `pnpm test:bun` — runs the whole jest suite under Bun, one process per
  file (jest-equivalent module isolation). `pnpm test:bun:contract` — Bun-only
  client-contract tests (`tests-bun/`).

## Testing

- `LINKGRESS_TEST_DRIVER=pg|postgres|bun` (default `pg`) selects the driver
  built by `tests/utils/test-database.ts`.
- New `expectToReject()` helper replaces all `expect(...).rejects.toThrow`
  call sites (17 across 12 files): the `.rejects`/`.resolves` matchers hang or
  misreport on DB-backed promises under bun:test and wedge the rest of the
  file.
- BunClient jest mocks made faithful to real Bun.SQL result shapes (result
  sets are true arrays carrying `command`/`count`).
- `debug/bench-drivers.ts` — serial round-trip micro-benchmark of the three
  clients under Node or Bun.
