# v0.4.38

## Feature: SqlFragment values in `upsertBulk` — compute expressions inside the upsert

### What

`db.<table>.upsertBulk()` values may now be **SqlFragments** (magic SQL strings), not
just plain literals — mirroring what `UpdateData` already allowed for `update()`. A
fragment renders as a SQL expression inside the `VALUES` tuple, with its interpolated
values parameterized, so e.g. a **scalar subquery** is computed by the INSERT itself
and flows into the `DO UPDATE` arm via `EXCLUDED."col"`. A read-fold-write cycle
(aggregate something, then upsert the result) collapses into **one round trip**:

```typescript
import { sql } from 'linkgress-orm';

// One statement: fold the user's paid spend AND upsert the accumulator row.
const rows = await db.userSpends.upsertBulk(
  [
    {
      userId,
      totalSpend: sql<number>`(
        SELECT COALESCE(SUM("total_price"), 0)::int
        FROM "order"
        WHERE "user_id" = ${userId} AND "status" = ${OrderStatus.Paid}
      )`,
      modifiedAt: new Date(),
    },
  ],
  {
    primaryKey: 'userId',
    updateColumns: ['totalSpend', 'modifiedAt'],
  }
).returning(s => ({ totalSpend: s.totalSpend }));
// INSERT INTO "user_spend" ("user_id", "total_spend", "modified_at")
// VALUES ($1, (SELECT COALESCE(SUM("total_price"), 0)::int FROM "order" WHERE "user_id" = $2 AND "status" = $3), $4)
// ON CONFLICT ("user_id") DO UPDATE SET "total_spend" = EXCLUDED."total_spend", "modified_at" = EXCLUDED."modified_at"
// RETURNING "total_spend"
```

### Semantics

- Fragments must be **self-contained SQL**: they render inside a `VALUES` tuple where
  no table alias is in scope, so entity column references are not resolvable there
  (interpolate literals / `sql.raw` / nested fragments / subqueries instead).
- Interpolated values bind as ordinary `$n` parameters, correctly renumbered across
  mixed rows (fragment rows and plain rows can share one statement).
- Fragment values **bypass the column's type mapper** (the fragment IS the SQL); plain
  values in the same row keep going through `toDriver` as before.
- Aggregate-only scalar subqueries (`SELECT SUM(...)`) always yield exactly one row,
  so the INSERT arm still materializes on an empty source (`SUM → NULL → COALESCE`).

### New API surface

- `UpsertData<TEntity>` — exported type alias of `UpdateData<TEntity>`;
  `upsertBulk(values: UpsertData<TEntity>[], config?)` now accepts it (every existing
  `InsertData` call site remains assignable — fully backward compatible).
