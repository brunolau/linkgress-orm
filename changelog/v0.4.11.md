## Log level support for custom logger

### Custom logger now receives log level

The `logger` function in `QueryOptions` now accepts an optional second parameter indicating the log level (`'info' | 'warn' | 'error' | 'debug'`). This allows custom loggers to route messages appropriately (e.g. to different log transports or severity levels).

```typescript
const db = new AppDatabase(client, {
  logQueries: true,
  logger: (message, level) => {
    switch (level) {
      case 'error': console.error(message); break;
      case 'warn':  console.warn(message);  break;
      default:      console.log(message);   break;
    }
  }
});
```

New exported type: `LogLevel = 'info' | 'warn' | 'error' | 'debug'`

### Logger propagated to schema manager and migrations

The custom logger is now passed through to `DbSchemaManager`, `MigrationRunner`, and `EnumMigrator` — previously these components always used `console.log` regardless of the configured logger.

All `console.log` calls in the migration and schema management code have been replaced with `this.logger(...)` calls with appropriate log levels:
- SQL queries, execution times, and trace summaries → `'debug'`
- SQL errors → `'error'`
- Skipped operations and circular dependency warnings → `'warn'`
- General progress messages → default (no level)

### Files Changed

- `src/entity/db-context.ts` — `LogLevel` type; `QueryOptions.logger` signature updated to `(message: string, level?: LogLevel) => void`; `TimeTracer` and `QueryExecutor` pass log levels; `getSchemaManager()` and `getMigrationRunner()` forward logger
- `src/index.ts` — exports `LogLevel`
- `src/migration/db-schema-manager.ts` — all `console.log` replaced with `this.logger()` with appropriate levels
- `src/migration/enum-migrator.ts` — accepts and uses custom logger
- `src/migration/migration-runner.ts` — accepts and uses custom logger
- `src/migration/migration.interface.ts` — `MigrationConfig.logger` signature updated
