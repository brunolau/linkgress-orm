# v0.4.45

## Filter-preserving joins — `joinFilter()` / `leftJoinFilter()`

Joins usable as pure row FILTERS: unlike `innerJoin`/`leftJoin` they take no selector, so the entity/selection shape is preserved and the builder stays chainable (`where`/`orderBy`/`limit`/`count`/`asSubquery`). Available on `IEntityQueryable`, `DbEntityTable` (directly off the table) and `SelectQueryBuilder`.

```typescript
// Scope predicate hopping an N:1 FK in ONE query level — no nested IN:
const scoped = db.orderItems
	.joinFilter(
		db.invoicingPartnerData,
		(oi, d) => eq(d.id, oi.invoicingPartnerDataId!),
		(_oi, d) => inArray(d.invoicingPartnerId, partnerIds),
	);

// Anti-join (rows with NO matching right row):
const usersWithoutPosts = db.users
	.leftJoinFilter(db.posts, (u, p) => eq(p.userId, u.id), (_u, p) => isNull(p.id));
```

The optional third callback contributes an extra WHERE predicate with the same `(left, right)` arguments. Only the right table's COLUMNS are addressable (no navigations). Joining a 1:N side duplicates left rows — use `exists()`/`inSubquery()` for semi-join semantics there.

Motivation: at 2M-users/3M-items scale the flat JOIN shape measured 1.3–5× faster than the equivalent nested `IN (SELECT … WHERE fk IN (SELECT …))` scope predicate (the second IN level costs a separate materialization and degrades row estimates; worst on long-tail partners: 9.0ms → 1.9ms for a scoped order page).

## Correlated standalone subqueries in `exists()` — pinned + guarded

Cross-table correlation of a STANDALONE builder inside `exists()`/`notExists()` (`exists(db.posts.where(p => eq(p.userId, u.id)).select(...).asSubquery())`) is now covered by tests as officially supported — including from inside navigation-collection lambdas (correlated collection counts). Outer field references render with the outer alias and the subquery emits its own FROM, so no dangling-alias problem arises (unlike WHERE-only navigations in correlated contexts).

SAME-TABLE self-correlation (inner and outer over one table) cannot render correctly — both sides share the alias and the correlation silently binds to the inner row (wrong results, no SQL error). Builders now carry a chain identity (`chainId`) baked into root field refs; `asSubquery()` detects a foreign-chain reference over its own table and THROWS a descriptive error instead of returning wrong rows.
