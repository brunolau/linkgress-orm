## Fix: `.min()` / `.max()` / `.sum()` ignore nav-property joins required by WHERE

### What

`.min()`, `.max()`, and `.sum()` on the outer query now emit the JOINs needed by nav-property column references in the WHERE clause, so filters like `where(p => eq(p.segment!.active, true))` work the same as they already do for `.toList()`, `.first()`, and `.count()`.

```typescript
// Previously threw `missing FROM-clause entry for table "segment"` —
// now compiles to the expected LEFT JOIN + WHERE form.
const minId = await db.segmentProducts
  .where(p => eq(p.segment!.active, true))
  .select(p => ({ id: p.id }))
  .min(p => p.id);
```

### Why it failed before

`SelectQueryBuilder` has four parallel SQL builders:

| Builder | Used by | Calls `detectAndAddJoinsFromCondition` |
|---|---|---|
| `buildQuery` | `toList`, `first`, `firstOrDefault` | yes (line 4230) |
| `buildAggregateQuery` | `count`, `exists` | yes (line 5574) |
| `buildAggregationQuery` | `min`, `max`, `sum` | **no** |
| `update` / `delete` | mutations | yes |

The three working builders walked the WHERE clause's `FieldRef`s, collected each `__tableAlias` / `__navigationAliases`, and emitted the corresponding JOIN before the WHERE was rendered. `buildAggregationQuery` skipped that step, so a query like `where(p => eq(p.segment!.active, true)).min(p => p.id)` generated:

```sql
SELECT MIN("segmentProducts"."id") AS result
FROM "gobo226_segment_products"
WHERE "segment"."active" = $1   -- "segment" never joined
```

PostgreSQL responded with `missing FROM-clause entry for table "segment"`, unless an ambient `segment` table existed in the search_path — in which case the planner cross-joined it silently and the aggregate returned nonsense.

### How the fix routes it

`buildAggregationQuery` now mirrors the other three builders: before rendering the WHERE clause, it runs `detectAndAddJoinsFromCondition(this.whereCond, navJoins)` and then, after the manual-join loop, emits one `LEFT JOIN` / `INNER JOIN` per detected nav join using the relation's `isMandatory` flag. The join-clause shape (alias quoting, composite-key handling, `formatJoinValue`) is copy-aligned with the `buildAggregateQuery` block so the two stay in sync.

```sql
SELECT MIN("segmentProducts"."id") AS result
FROM "gobo226_segment_products"
LEFT JOIN "gobo226_segments" AS "segment"
  ON "gobo226_segment_products"."segment_id" = "segment"."id"
WHERE "segment"."active" = $1
```

### Files Changed

- `src/query/query-builder.ts` — `buildAggregationQuery`: detect nav-joins from the WHERE clause, then emit them onto the FROM clause before the WHERE renders.
- `tests/queries/nav-where-left-join-filter.test.ts` — new test file covering all five outer-query shapes (`toList`, `count`, `first`, `min`, `max`) against an `isMandatory=false` (LEFT JOIN) nav relation, plus a compound-AND variant and a regression guard for mandatory (INNER JOIN) relations. The `min`/`max` cases are the ones that flip from PostgreSQL error → green; the others are guards against regressing the already-working paths.
