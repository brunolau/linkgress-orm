## Feature: CTE references inside `` sql`` `` templates ŌĆö `DbCte.as()`

### What

Raw `` sql`` `` fragments that needed to reference a CTE previously had to
hand-write the identifiers as strings (`"post_stats"."views"`, `FROM "post_stats"
AS "ps"`) ŌĆö untyped, easy to typo, and silently wrong if the CTE was renamed.

`DbCte.as(alias?)` now returns a **typed** reference that renders the right
identifier wherever it is interpolated, mirroring SQL's `FROM cte AS alias`:

- interpolating a **column** (`${ps.views}`) renders a qualified column
  identifier ŌĆö `"ps"."views"`;
- interpolating the **ref itself** (`${ps}`) renders the FROM-clause table
  reference ŌĆö `"post_stats" AS "ps"`.

```typescript
const stats = cteBuilder.with('post_stats', db.posts.select(p => ({
  postId: p.id,
  views: p.views,
  authorId: p.userId,
})));

const ps = stats.cte.as('ps');

sql`(SELECT COALESCE(SUM(${ps.views}), 0) FROM ${ps} WHERE ${ps.authorId} = ${u.id})`
// ŌåÆ (SELECT COALESCE(SUM("ps"."views"), 0) FROM "post_stats" AS "ps" WHERE "ps"."authorId" = "users"."id")
```

Calling `.as()` without an alias qualifies columns by the CTE name and renders
the table ref as just `"post_stats"` ŌĆö no redundant `AS` is emitted.

The returned `CteTableRef<TColumns>` is strongly typed: column properties are
derived from the CTE's selection, so they autocomplete and a renamed/removed CTE
column is a compile error rather than a broken string.

### Behaves like a real FieldRef

`.as()` mirrors the internal CTE mock row (`createMockRowForCte`), so the refs
carry the same metadata and work anywhere a `FieldRef` is accepted:

- **custom type mappers** are preserved (`getMapper`), so a CTE column declared
  with `mapWith` still maps correctly through a fragment;
- **aggregation-array** metadata (`__isAggregationArray` /
  `__innerSelectionMetadata`) is preserved, so correlated JSON aggregates render
  identically to hand-written SQL.

The table ref is recognized distinctly from a column ref ŌĆö it is never mistaken
for a `FieldRef` nor bound as a query parameter ŌĆö and parameter numbering stays
correct when CTE params, fragment literals, and outer `where` params are mixed in
one query.

### How

`SqlFragment.buildSql()` gained a branch that detects a CTE table reference and
emits the FROM-clause identifier. The check is **duck-typed** on marker
properties (`__isCteTableRef` / `__cteName` / `__tableAlias`) rather than an
`instanceof`, to avoid a circular import between `conditions.ts` and
`cte-builder.ts`.

### Files Changed

- `src/query/cte-builder.ts` ŌĆö new exported `CteTableRef<TColumns>` type and
  `DbCte.as(alias?)`, which builds the ref (marker props + a typed `FieldRef` per
  CTE column, carrying mapper / aggregation-array metadata).
- `src/query/conditions.ts` ŌĆö `SqlFragment.isCteTableRef()` (duck-typed) and a
  `buildSql()` branch rendering `"cte" AS "alias"` (or `"cte"` when unaliased).
- `src/index.ts` ŌĆö export the `CteTableRef` type.
- `tests/queries/cte-sql-template-refs.test.ts` ŌĆö **new**: 6 tests ŌĆö SQL
  rendering (aliased column + FROM table ref; unaliased qualified by CTE name;
  correlated `json_build_array` aggregate byte-identical to hand-written SQL;
  table ref not mistaken for a FieldRef nor parameter-bound) and execution
  (correlated scalar subquery over an aliased CTE with `mapWith`; parameter
  numbering across CTE params + fragment literals + outer where params).
- `tests/migration/temp_migrations/*.ts` ŌĆö removed two accidentally-committed
  fixtures that `manual-migration.test.ts` generates into that directory at run
  time.
