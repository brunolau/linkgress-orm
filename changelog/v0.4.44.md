# v0.4.44 (fix: BunClient no longer breaks downstream bundlers — hidden `bun` specifier)

`BunClient` is re-exported from the package entry, so any bundler that pulls in
`linkgress-orm` walks into `dist/database/bun-client.js` — even in an app that
only uses `PgClient` / `PostgresClient` and never touches Bun. That file
resolved Bun's SQL constructor through a literal `require('bun')` fallback. The
`bun` module only exists inside the Bun runtime, so build-time bundlers that
statically resolve literal `require()` specifiers fail on it:

```
[vite]: Rolldown failed to resolve import "bun" from
".../linkgress-orm/dist/database/bun-client.js".
```

(Reported against Vite 8 / Rolldown 1.1.4 building a frontend admin bundle;
Vite escalates Rolldown's `UNRESOLVED_IMPORT` warning into a hard build error.)

**Fix.** The `bun` module specifier is now assembled at runtime from an
expression bundlers won't constant-fold, so static analysis can't see it:

```typescript
const bunModuleId = ['b', 'u', 'n'].join('');
const { SQL } = require(bunModuleId);
```

The `Bun` **global** remains the primary path — under Bun, `globalThis.Bun.SQL`
is used and the `require` branch is never reached. The `require` fallback only
runs in the (rare) case of Bun without the global, and under Node it throws and
is caught by the constructor exactly as before. Runtime behavior is unchanged;
only the build-time surface changed.

Verified against Rolldown 1.1.4 with Vite's warning-to-error escalation: a
plain string literal and `'b' + 'un'` are both constant-folded and re-trigger
the error (the literal panicked Rolldown outright); the array-`join` form
bundles with zero warnings. The real compiled `dist/database/bun-client.js`
bundles cleanly, and both runtime paths (Bun global present / Node fallback)
behave as before.

> **Consumers pinning a specific version must upgrade to `0.4.44`** to pick up
> the fix. As an interim workaround without upgrading, add `bun` to the app's
> `build.rollupOptions.external` (Vite) — but upgrading is preferred, since it
> removes the unresolvable import at the source.
