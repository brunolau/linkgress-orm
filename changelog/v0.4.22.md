## Feat: SQL expressions in `update()` values + `jsonbMerge` helper

### What

`update()` now accepts `SqlFragment` values, so an UPDATE column can be set from a SQL expression instead of a literal. This unlocks atomic in-place mutations (counters, JSONB merges, conditional updates) that previously required a raw query.

```typescript
// In-place arithmetic
await db.users
  .where(u => eq(u.id, userId))
  .update({ age: sql`age + 1` });

// Atomic JSONB merge — no read-modify-write race
await db.orderItems
  .where(p => eq(p.id, id))
  .update(p => ({ integrationInfo: jsonbMerge(p.integrationInfo, patch) }));
```

Two new pieces support this:

1. **`update()` accepts a function form** — `update(p => ({...}))` where `p` is the column proxy, so `SqlFragment` values can reference the row's own columns (e.g. `p.integrationInfo`). The plain `update({...})` form still works for `SqlFragment`s that don't need a column reference (`sql\`age + 1\``).
2. **`jsonbMerge(target, patch)`** — convenience wrapper for `COALESCE(target, '{}'::jsonb) || $patch::jsonb`, the standard atomic JSONB merge pattern. Safe under concurrent writes because `||` is evaluated per-row by PostgreSQL.

`coalesce()` also gained variadic support (`coalesce(a, b, c, ...)`) — previously only two arguments were accepted.

### Why it failed before

The previous `update()` implementation iterated `Object.entries(data)` and bound every value as a `$N` parameter:

```ts
setClauses.push(`"${config.name}" = $${paramIndex++}`);
values.push(value);
```

A `SqlFragment` passed in here was JSON-serialised by the driver as a literal `{}` object instead of being emitted as SQL. The only way to update a column with a SQL expression was to drop into `db.query(...)`.

For JSONB specifically, the absence of an in-place merge forced an unsafe read-modify-write pattern (`SELECT … then UPDATE …`) that loses interleaving writes under concurrency.

### How the fix routes it

- `update()` (both `DbEntityTable.update` and `SelectQueryBuilder.update`) now detects `value instanceof SqlFragment`. When found, it builds the fragment's SQL with a shared `SqlBuildContext` so the fragment's own bound parameters merge into the UPDATE's `values` array with continuous `$N` numbering, then emits `"col" = <fragmentSql>` instead of `"col" = $N`.
- The function form invokes the user's callback with the table's column proxy (`table.props()` for `DbEntityTable`, `_createMockRow()` for `SelectQueryBuilder`). Column references inside `SqlFragment`s — `p.integrationInfo` — resolve to `FieldRef`s against the actual table at build time.
- `jsonbMerge` is a thin `SqlFragment` template: `COALESCE($target, '{}'::jsonb) || $patch::jsonb`. The `COALESCE(target, '{}')` handles `NULL` columns; the patch is cast to `jsonb` so plain JS objects bind cleanly.
- A new `UpdateData<TEntity>` type widens each column to `T | SqlFragment<T>` so the SqlFragment form type-checks. `InsertData<TEntity>` is unchanged (inserts still take plain values).
- `coalesce()` was extended with an overload accepting `(value1, value2, ...rest)` and its body now builds the SQL parts dynamically (`'COALESCE('`, `', '` × n, `')'`) instead of a hard-coded 3-element template.

### Example SQL

```sql
-- update({ age: sql`age + 10` })
UPDATE "users" SET "age" = age + 10 WHERE "id" = $1

-- update(p => ({ metadata: jsonbMerge(p.metadata, { baz: 'qux' }) }))
UPDATE "users" SET "metadata" = COALESCE("metadata", '{}'::jsonb) || $1::jsonb WHERE "id" = $2
```

### Files Changed

- `src/entity/db-column.ts` — new `UpdateData<TEntity>` type; column values widened with a forward-declared `SqlFragment` marker (the real class lives in `query/conditions`, kept out to avoid a circular import).
- `src/entity/db-context.ts` — `DbEntityTable.update` and `IEntityQueryable.update` / `EntitySelectQueryBuilder.update` signatures accept `UpdateData<TEntity> | ((row) => UpdateData<TEntity>)`; runtime resolves the function form with `table.props()` and inlines any `SqlFragment` values via a shared `SqlBuildContext`.
- `src/query/query-builder.ts` — `SelectQueryBuilder.update` mirrors the above (uses `_createMockRow()` for the proxy), inlines `SqlFragment` values, and continues to apply column `toDriver` mappers for literal values.
- `src/query/conditions.ts` — `coalesce()` now variadic (2+ args); new `jsonbMerge(target, patch)` helper.
- `src/index.ts` — re-exports `jsonbMerge` and `UpdateData`.
- `tests/mutations/insert-update-delete.test.ts` — five new tests: literal `SqlFragment` in update, `SqlFragment` with bound params, `jsonbMerge` round-trip preserving prior keys, `jsonbMerge` against a `NULL` column, and variadic `coalesce` with 3 arguments.
