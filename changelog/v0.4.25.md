## Feat: `UnionQueryBuilder.asSubquery()` — use a UNION as a subquery in WHERE / SELECT / JOIN

### What

A `UnionQueryBuilder` can now be wrapped as a `Subquery` and embedded directly inside an outer query — mirroring `SelectQueryBuilder.asSubquery()` but composing multiple SELECT legs into a single subquery expression. Pairs with `inSubquery(...)` / `exists(...)` so an outer filter can target the merged row set in one roundtrip.

```typescript
// All users whose id appears in EITHER side of a friend relation,
// resolved in a single SQL statement.
const friendIds = db.userRelations
  .where(r => eq(r.parentId, userId))
  .select(r => r.slaveId)
  .unionAll(
    db.userRelations
      .where(r => eq(r.slaveId, userId))
      .select(r => r.parentId),
  )
  .asSubquery('array');

const friends = await db.users
  .where(u => inSubquery(u.id, friendIds))
  .select(u => ({ id: u.id, name: u.username }))
  .toList();
```

### Why it didn't exist before

`UnionQueryBuilder.toList()` could only run the union as a top-level query. To filter another query by "membership in a union of N row sources" the caller had to issue each leg as a separate query and merge the id sets in application code — extra roundtrips and no SQL-level composition with the outer query's WHERE / JOIN.

The blocker wasn't shape — it was parameter numbering. The existing `buildSql()` allocated a fresh `SqlBuildContext` with `paramCounter: 1` every call. Embedded as a subquery, this would restart `$1, $2, …` indices inside the outer statement, colliding with the outer query's own placeholders.

### How the fix routes it

`buildSql()` is now overloaded:

- `buildSql(): { sql, params }` — top-level form, unchanged. Allocates a fresh context.
- `buildSql(outerContext): string` — nested form. Reuses the outer query's `paramCounter` and `params` array so `$1, $2, …` numbering chains continuously across the composite statement, and returns the bare SQL fragment (no params object — the outer context already owns the params).

`asSubquery<TMode>(mode)` wraps a callback that invokes the nested form into a `Subquery<TSelection, TMode>`. The three modes mirror `SelectQueryBuilder.asSubquery`:

- `'table'` (default) — full rows, suitable for `FROM (...)`
- `'array'` — single-column list, for `inSubquery(...)`
- `'scalar'` — single value, for direct comparison or projection

### Files Changed

- `src/query/union-builder.ts` — overload `buildSql()` to accept an outer `SqlBuildContext` (returns bare SQL when nested, keeps the existing `{ sql, params }` shape for the top-level call); add `asSubquery<TMode>()` that wraps the nested form in a `Subquery`.
- `tests/queries/union-subquery.test.ts` — new test file: `inSubquery` against a 2-leg UNION ALL, a 3-leg variant, and a regression test for parameter-counter chaining (literal params in the outer WHERE plus more inside each union leg, asserting no `$`-index collisions or `bind message has N parameters` errors).
