## Bug Fixes

### Fixed `eq()`/`ne()` generating invalid SQL with `undefined` values

When passing `undefined` as a value to `eq()` or `ne()` (e.g. `eq(p.userId, args.userId)` where `args.userId` is `undefined`), the ORM generated broken SQL like `"user_id" = AND "is_current" = $1` — a bare operator with no right-hand side.

Now `undefined` is treated the same as `null`:
- `eq(field, undefined)` → `"field" IS NULL`
- `ne(field, undefined)` → `"field" IS NOT NULL`

### Added guard for other comparison operators with `undefined` values

Other binary operators (`gt`, `lt`, `gte`, `lte`, `like`) now throw a clear error when given `undefined` instead of silently generating broken SQL. The error message directs developers to use `eq()`/`ne()` which handle `undefined` as NULL semantics.

### Files Changed

- `src/query/conditions.ts` - `EqComparison.buildSql()` and `NeComparison.buildSql()` handle `undefined` as `IS NULL`/`IS NOT NULL`; `WhereComparisonBase.buildSql()` throws on `undefined` for non-unary operators
