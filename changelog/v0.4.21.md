## Fix: `hasDefaultValue('')` produces invalid SQL

### What

Passing an empty string to `hasDefaultValue('')` now correctly emits `DEFAULT ''` in the generated DDL instead of producing a syntax error.

```typescript
entity.property(e => e.label)
  .hasType(varchar('label', 100))
  .isRequired()
  .hasDefaultValue('');
```

### Why it failed before

The default-value formatter passes string values through verbatim (the documented convention is that string defaults are pre-quoted literals like `` `'tech'` `` or SQL expressions like `'NOW()'`). For the empty string, "verbatim" yields nothing, so the generated DDL was:

```sql
CREATE TABLE "empty_default_items" (
  "id"    integer GENERATED ALWAYS AS IDENTITY NOT NULL,
  "label" varchar(100) NOT NULL DEFAULT ,
  PRIMARY KEY ("id")
)
```

PostgreSQL rejected this with `syntax error at or near ","`.

### How the fix routes it

`formatDefaultValue` (`db-schema-manager.ts`) and `formatDefault` (`migration-scaffold.ts`) now special-case the empty string and emit `''` (the SQL empty-string literal). All other string inputs continue to pass through unchanged, so pre-quoted literals (`` `'tech'` ``) and SQL expressions (`'NOW()'`, `'CURRENT_TIMESTAMP'`) are unaffected.

### Files Changed

- `src/migration/db-schema-manager.ts` — `formatDefaultValue` returns `"''"` when the string value is empty.
- `src/migration/migration-scaffold.ts` — `formatDefault` returns `"''"` when the string value is empty.
- `tests/schema/has-default-empty-string.test.ts` — new integration test: creates a table with `hasDefaultValue('')`, asserts `information_schema.columns.column_default` is the empty-string literal, and verifies `INSERT … DEFAULT VALUES` produces `''`.
