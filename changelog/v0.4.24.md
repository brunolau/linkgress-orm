## Fix: constant FK predicates on `hasMany` navigations now fire in collection projections

### What

Navigation properties that extend `withForeignKey` / `withPrincipalKey` past the column-equality pair with literal values now have those literals emitted on every collection projection — across all three strategies (`lateral`, `cte`, `temptable`).

```typescript
// Intent: every projection of parent.children is restricted to is_current = TRUE
entity.hasMany(e => e.children, () => Child)
  .withForeignKey(c => [c.parentId, c.isCurrent])
  .withPrincipalKey(p => [p.id,     true]);
```

Before this fix, `parent.children` projected via `.select(...).toList()` leaked the closed (SCD2 historical) rows — the literal `is_current = TRUE` predicate was silently dropped at the collection layer. Direct JOINs (when `.children` was navigated through a WHERE/SELECT, not projected as a collection) already honored the literal predicate; only the collection-projection paths were broken.

### Why it failed before

The `__LIT:<value>` marker mechanism for literal key parts was already in place (`join-utils.ts`, `formatJoinValue`), and direct nav-joins through WHERE/SELECT already used it. But `CollectionQueryBuilder` was constructed with a single-column `foreignKey: string` parameter that predated composite/literal FKs. At construction time it kept `foreignKeys[0]` and discarded everything else, so every collection strategy emitted a hard-coded correlation:

```sql
-- LATERAL
WHERE "lateral_0_children"."parent_id" = "parent"."id"
-- MISSING: AND "lateral_0_children"."is_current" = true
```

The same shape existed in CTE inner SELECTs and the temptable strategy's trailing WHERE — each path emitted exactly one column-equality clause and ignored the rest of the FK arrays.

### How the fix routes it

The fix threads the full `foreignKeys[]` / `matches[]` arrays end-to-end, then introduces two helpers that match each strategy's correlation shape.

**Surface — `CollectionQueryBuilder`:** new constructor params `foreignKeys?: string[], matches?: string[]`, stored on the instance and defaulted to `[foreignKey]` / `['id']` for backward compatibility. The arrays propagate through `.where()`-cloned builders and into `getAggregationConfig()`, which now passes them down to the strategy layer via two new optional fields on `CollectionAggregationConfig`: `foreignKeys?: string[]` and `matches?: string[]` (the original single `foreignKey` is retained but marked `@deprecated`).

**Propagation:** every site that constructs a `CollectionQueryBuilder` from `relConfig` — across `src/query/query-builder.ts` (5 sites), `src/query/grouped-query.ts` (2 sites), and `src/entity/db-context.ts` (1 site) — now forwards `relConfig.foreignKeys` and `relConfig.matches` as the trailing args.

**Two helpers in `src/query/join-utils.ts`** reflect that LATERAL and CTE/temptable correlate the parent differently:

- `buildCollectionCorrelationWhere(fkAlias, sourceAlias, foreignKeys, matches)` — iterates the arrays in lock-step and emits `fkAlias.fk = sourceAlias.match` for every pair, routing literal markers through `formatJoinValue` so `__LIT:true` renders as `true`. Used by **LATERAL**, which has both parent and child tables in scope inside the subquery.
- `buildLiteralOnlyPredicates(targetAlias, foreignKeys, matches)` — emits only the pairs where the principal side is a literal; column=column pairs are skipped. Used by **CTE** and **temptable**, which materialize the parent correlation through `parent_id` grouping (CTE) or a `parent_ids` temp table (temptable), so the parent table is not in scope inside the inner SELECT. Re-emitting `child.fk = parent.match` there would reference a non-existent alias; only the literal filters need to be ANDed in.

**Strategy wiring:**

- **LATERAL** (5 inner-SELECT sites in `lateral-collection-strategy.ts`): a new private `buildParentCorrelation()` replaces the hard-coded `"fk"."col" = "source"."id"` everywhere. Falls back to the legacy single-column form when the array is absent.
- **CTE** (5 inner-SELECT sites in `cte-collection-strategy.ts`): a new private `composeInnerWhere()` ANDs the user's already-rewritten WHERE clause together with the literal predicates and emits a single `WHERE` clause.
- **Temptable** (5 inner-SELECT sites in `temptable-collection-strategy.ts`): a new private `buildAdditionalWhere()` returns ` AND <preds>` to splice into the existing `WHERE temp.id IN (...)` clause.

### Example SQL diff (LATERAL)

```sql
-- Before (broken — literal predicate dropped)
SELECT json_build_object(...)
FROM "cfk_children" "lateral_0_children"
WHERE "lateral_0_children"."parent_id" = "cfk_parents"."id"

-- After
SELECT json_build_object(...)
FROM "cfk_children" "lateral_0_children"
WHERE "lateral_0_children"."parent_id"  = "cfk_parents"."id"
  AND "lateral_0_children"."is_current" = true
```

For CTE/temptable the predicate is ANDed into the inner-SELECT's WHERE rather than the LATERAL correlation, but the resulting target-row filter is the same.

### Backward compatibility

- `CollectionAggregationConfig.foreignKey` is retained (marked `@deprecated`) and equals `foreignKeys[0]`. All strategies fall back to the single-column form when the arrays are absent.
- `CollectionQueryBuilder` constructor's new array params are optional; existing callsites that don't pass them are unaffected.
- `matches` defaults to `['id']` to preserve prior behavior for simple PK-on-the-parent navigations.

### Files Changed

- `src/query/join-utils.ts` — added `buildCollectionCorrelationWhere()` (composite-aware full correlation, for LATERAL) and `buildLiteralOnlyPredicates()` (literal-only filter, for CTE/temptable).
- `src/query/collection-strategy.interface.ts` — added `foreignKeys?: string[]` and `matches?: string[]` to `CollectionAggregationConfig`; marked the existing `foreignKey` as `@deprecated`.
- `src/query/query-builder.ts` — `CollectionQueryBuilder` gains `foreignKeys`/`matches` constructor params and instance fields; `.where()` clones propagate them; `getAggregationConfig()` writes them onto the aggregation config; local `buildCTE()` correlation switches to `buildCollectionCorrelationWhere`. Five `CollectionQueryBuilder` construction sites updated to forward `relConfig.foreignKeys`/`matches`.
- `src/query/grouped-query.ts` — two construction sites updated.
- `src/entity/db-context.ts` — one construction site updated.
- `src/query/strategies/lateral-collection-strategy.ts` — new `buildParentCorrelation()` replaces the hard-coded correlation at all 5 inner-SELECT sites.
- `src/query/strategies/cte-collection-strategy.ts` — new `composeInnerWhere()` ANDs literal predicates into the inner WHERE at all 5 sites.
- `src/query/strategies/temptable-collection-strategy.ts` — new `buildAdditionalWhere()` ANDs literal predicates into the trailing WHERE at all 5 sites.
- `tests/queries/navigation-constant-fk-projection.test.ts` — new test file covering all three strategies plus a 3-column FK variant `[parentId, isCurrent, generation] / [id, true, 2]` and a regression for `.where()` composing (not replacing) the literal predicates. Asserts both the projected row set (semantic) and the emitted SQL contains `"is_current" = true` (structural).
