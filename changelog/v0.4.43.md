# v0.4.43 (feat: declarative extended statistics — `entity.hasStatistics()`)

PostgreSQL's planner has no statistics for expression predicates, so a WHERE
like `(flags & 1::smallint) = 0` falls back to the default operator
selectivity and estimates ~1 row regardless of the real distribution. When
such a predicate sits inside a scope subquery, the misestimate flips the
outer join strategy: measured on a real workload (a resort-scope subquery
matching 127 of 274 rows, driving a 2k-row list), the bad estimate produced a
per-row nested loop at 3.0 ms / 3,015 shared-hit buffers where the correct
plan is a hashed semi-join at 0.70 ms / 110 buffers. The fix is a
`CREATE STATISTICS` object on the expression — which until now had to live in
hand-written migrations, invisible to the model. Extended statistics are now
first-class schema items, declared next to the indexes they complement:

```typescript
// Univariate expression statistics (planner selectivity for a bitmask test):
entity.hasStatistics('stx_resort_admin_visible_flags')
  .withExpression('("flags" & 1::smallint)');

// Multivariate statistics over correlated columns, with explicit kinds:
entity.hasStatistics('stx_city_zip', e => [e.city, e.zip])
  .withKinds('ndistinct', 'dependencies');
```

- **`hasStatistics(name, selector?)`** mirrors `hasIndex`: plain columns come
  from the same selector-proxy mechanism; expressions the `ix*` helpers cannot
  express (infix operators, casts) are appended as raw SQL via
  **`.withExpression(...)`** — the same escape-hatch idiom `IndexBuilder`
  already has. **`.withKinds('ndistinct' | 'dependencies' | 'mcv')`** narrows
  multivariate kinds; a single-expression declaration produces univariate
  EXPRESSION statistics (PostgreSQL's `stxkind = 'e'`).
- **Schema manager integration**: `ensureCreated()` builds declared objects
  after indexes; `analyze()` / `migrate()` reconcile per table **by name
  only** — a missing object is planned as a `create_statistics` operation,
  and a same-named object is never diffed, dropped, or rebuilt (rename the
  object to change its definition). Creation is
  `CREATE STATISTICS IF NOT EXISTS …` followed by `ANALYZE <table>`, so the
  new statistics affect plans immediately instead of waiting for autovacuum.
- **Fail-fast validation**: an empty `ON` list, or `.withKinds()` combined
  with a single expression entry (PostgreSQL rejects a kinds list on
  univariate expression statistics), throws a descriptive error at
  create time rather than surfacing as a cryptic server error mid-migration.
- **Migration scaffold**: `create_statistics` operations render into file
  migrations — UP emits the `CREATE STATISTICS IF NOT EXISTS` + `ANALYZE`
  pair, DOWN emits `DROP STATISTICS IF EXISTS`.
- **Plumbing / exports**: `TableSchema.statistics`,
  `TableBuilder.withStatistics()`, and the exported types
  `StatisticsMetadata` (entity metadata) / `StatisticsDefinition` (table
  schema). SQL builders live in `src/migration/statistics-sql.ts`, shared by
  the live migrate path and the scaffold so both emit identical statements.

**Operational note — expression form must match the plan tree.** Extended
statistics only apply when the declared expression tree equals the one in the
query. For a `smallint` column, `flagHasNone(col, 1)` binds the mask as
smallint, so the statistics must be declared as `(col & 1::smallint)`; an
integer-literal spelling (`(col & 1)`) creates statistics on the
`int4`-coerced expression, which never matches and silently does nothing.

**Internal**: `DbModelConfig.buildTables()` reconstructs every `TableBuilder`
in its foreign-key pass; statistics (like partitioning before them) are now
explicitly re-carried through that rebuild. Any future table-level
declaration must do the same or it silently disappears from the built schema.

## Testing

- `tests/schema/extended-statistics.test.ts` — live-DB coverage: univariate
  expression statistics via `ensureCreated()` (catalog `stxkind` contains
  `'e'`), multivariate kinds from selector columns (`'d'`/`'f'`), `migrate()`
  creating a missing object on an existing table and reporting in-sync
  afterwards (name-only reconciliation), fail-fast on kinds-with-single-
  expression, metadata capture, and canonical SQL emission of the shared
  builders.
