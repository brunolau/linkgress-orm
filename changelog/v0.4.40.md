## Fix: Back-fill new enum values into an existing enum during auto-migration

Contributed by [@matoslavv](https://github.com/matoslavv) in [#14](https://github.com/brunolau/linkgress-orm/pull/14).

### What

`analyze()` only emitted a `create_enum` operation when an enum type was
entirely absent — it never reconciled the **values** of an enum that already
existed. So adding a value to a `pgEnum` propagated to fresh databases (baked
into the initial `CREATE TYPE`) but was a silent no-op on any long-lived one.
Anything that cast to the enum afterwards then failed on those databases, e.g.
`invalid input value for enum <name>: "<new_label>"`.

```typescript
const status = pgEnum('order_status', ['draft', 'active', 'archived'] as const);
// 'archived' was appended to the model after the enum already existed:
//   - fresh DB:     enum has all three values (from CREATE TYPE)
//   - existing DB:  still only ('draft', 'active') → casts to 'archived' throw
// migrate() now detects the gap and adds the missing label automatically.
```

### How

Enum reconciliation now follows the same analyze → operation → execute path as
every other schema change:

- New `add_enum_value` `MigrationOperation`.
- `analyze()` reads an existing enum's labels from `pg_enum` and diffs them
  against the model, emitting `add_enum_value` for any missing labels. This is
  intentionally **add-only** — removing a value (`ALTER TYPE ... DROP VALUE`) is
  destructive, fails while any row still references the label, and callers often
  retain historical/stale labels on purpose — so pruning stays a manual
  migration.
- Executed in phase 1 (alongside `create_enum`, before any table/column that may
  reference the value) via `ALTER TYPE "<enum>" ADD VALUE IF NOT EXISTS '<v>'` —
  idempotent, safe under concurrent migrations, and usable by later operations
  since the migrate phases run in autocommit.

```
📦 Phase 1: ...
1. Add value(s) to ENUM "order_status" (archived)
```

### Files Changed

- `src/migration/db-schema-manager.ts` — `add_enum_value` operation type;
  existing-enum value diff in `analyze()`; phase-1 categorization;
  `executeAddEnumValues()` helper; `describeOperation()` label.
- `tests/schema/enum-add-value.test.ts` — regression tests: `analyze()` emits
  `add_enum_value` for a missing label, `migrate()` applies it, and a re-run is
  idempotent.
