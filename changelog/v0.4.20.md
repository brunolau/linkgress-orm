## Fix: nested collection reached through a reference chain emits intermediate joins

### What

A nested collection accessed through a chain of `hasOne` navigations inside another collection's selector — e.g. `cdc.discountCode!.discount!.discountProducts!` — now correctly emits the intermediate joins (`discount_codes`, `discounts`) in the outer collection's query so that the nested aggregation's correlation key resolves.

```typescript
await db.carts
  .select(c => ({
    cartId: c.id,
    items: c.cartItems!
      .select(it => ({ id: it.id }))
      .toList(),
    appliedCodes: c.cartDiscountCodes!
      .select(cdc => ({
        discountCodeId: cdc.discountCodeId,
        scopeProductIds: cdc.discountCode!.discount!.discountProducts!
          .select(dp => ({ id: dp.productId }))
          .toNumberList(),
      }))
      .toList(),
  }))
  .toList();
```

### Why it failed before

The nested `discountProducts` aggregation correlates on the chain's terminal PK (`discount.id`), but the outer `cartDiscountCodes` collection never emitted the joins needed to bring `discount_codes` and `discounts` into scope. Results:

- **LATERAL**: `missing FROM-clause entry for table "discount"` — the correlated subquery `WHERE "...".discount_id = "discount"."id"` referenced an alias that did not exist in the outer FROM.
- **CTE**: `column cart_discount_codes.id does not exist` — the nested-CTE join fell back to the outer collection's PK, but `cart_discount_codes` is a composite-key junction table with no `id` column.

The selection-walk in `detectNavigationJoins` stopped at `CollectionQueryBuilder` values and never harvested their `navigationPath`, so the outer builder was unaware that intermediate joins were needed.

### How the fix routes it

- `CollectionQueryBuilder.getNavigationPath()` exposes the already-captured reference chain (built when the proxy traverses `Reference → Reference → Collection`).
- `detectNavigationJoins` now pushes each step of a nested collection's `navigationPath` into the outer's join set (deduped by alias), so both CTE and LATERAL emit the intermediate joins directly from the outer's FROM.
- The CTE-strategy `nestedJoinClause` now targets the chain's terminal alias (e.g. `"discount"."id" = "cte_1".parent_id`) when a navigation path is present, and only falls back to `"<outerTable>"."id"` for direct-child collections.

### Example SQL diff

**Outer CTE before** (broken):

```sql
FROM "cart_discount_codes"
LEFT JOIN "cte_1" ON "cart_discount_codes"."id" = "cte_1".parent_id
```

**Outer CTE after**:

```sql
FROM "cart_discount_codes"
LEFT JOIN "discount_codes" "discountCode" ON "cart_discount_codes"."discount_code_id" = "discountCode"."id"
LEFT JOIN "discounts"      "discount"     ON "discountCode"."discount_id"             = "discount"."id"
LEFT JOIN "cte_1"                          ON "discount"."id"                         = "cte_1".parent_id
```

**Outer LATERAL before** (broken — inner correlated subquery references `"discount"."id"` with no matching FROM entry):

```sql
FROM "cart_discount_codes" "lateral_1_cartDiscountCodes"
WHERE "lateral_1_cartDiscountCodes"."cart_id" = "carts"."id"
```

**Outer LATERAL after**:

```sql
FROM "cart_discount_codes" "lateral_1_cartDiscountCodes"
LEFT JOIN "discount_codes" "discountCode" ON "lateral_1_cartDiscountCodes"."discount_code_id" = "discountCode"."id"
LEFT JOIN "discounts"      "discount"     ON "discountCode"."discount_id"                    = "discount"."id"
WHERE "lateral_1_cartDiscountCodes"."cart_id" = "carts"."id"
```

### Performance

Nav-path steps come directly from the proxy chain as fully-formed `NavigationJoin` objects, so the fix adds no new schema lookups, no second selector invocation, and no extra proxy rebuilds. Queries that do not cross a reference chain on the way to a nested collection are byte-identical to before.

### Files Changed

- `src/query/query-builder.ts`
  - `CollectionQueryBuilder.getNavigationPath()` — expose the captured reference chain to the outer builder.
  - `CollectionQueryBuilder.detectNavigationJoins` — when a selection value is a nested `CollectionQueryBuilder`, push its `navigationPath` into the outer's join set (deduped by alias); the prior `!(value instanceof CollectionQueryBuilder)` guard is no longer needed on the object-recursion branch.
  - `CollectionQueryBuilder.buildCTE` — the CTE `nestedJoinClause` now targets `<lastNavAlias>.<matchPK>` when the nested collection carries a navigation path, falling back to `<outerTable>.id` only for direct-child collections.
- `tests/queries/nested-to-number-list.test.ts` — the three previously-failing `impl.md exact reproduction (Cart schema)` cases (CTE, LATERAL, `firstOrDefault`) now pass; the rest of the file is unchanged.
